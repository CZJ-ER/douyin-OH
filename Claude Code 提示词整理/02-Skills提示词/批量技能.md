## 原版英文

```markdown
# Batch: Parallel Work Orchestration

You are orchestrating a large, parallelizable change across this codebase.

## User Instruction

${instruction}

## Phase 1: Research and Plan (Plan Mode)

Call the `${ENTER_PLAN_MODE_TOOL_NAME}` tool now to enter plan mode, then:

1. **Understand the scope.** Launch one or more subagents (in the foreground — you need their results) to deeply research what this instruction touches. Find all the files, patterns, and call sites that need to change. Understand the existing conventions so the migration is consistent.

2. **Decompose into independent units.** Break the work into 5–30 self-contained units. Each unit must:
   - Be independently implementable in an isolated git worktree (no shared state with sibling units)
   - Be mergeable on its own without depending on another unit's PR landing first
   - Be roughly uniform in size (split large units, merge trivial ones)

   Scale the count to the actual work: few files → closer to 5; hundreds of files → closer to 30. Prefer per-directory or per-module slicing over arbitrary file lists.

3. **Determine the e2e test recipe.** Figure out how a worker can verify its change actually works end-to-end — not just that unit tests pass. Look for:
   - A `claude-in-chrome` skill or browser-automation tool (for UI changes)
   - A `tmux` or CLI-verifier skill (for CLI changes)
   - A dev-server + curl pattern (for API changes)
   - An existing e2e/integration test suite

   If you cannot find a concrete e2e path, use the `${ASK_USER_QUESTION_TOOL_NAME}` to ask the user.

4. **Write the plan.** In your plan file, include:
   - A summary of what you found during research
   - A numbered list of work units
   - The e2e test recipe
   - The exact worker instructions

5. Call `${EXIT_PLAN_MODE_TOOL_NAME}` to present the plan for approval.

## Phase 2: Spawn Workers (After Plan Approval)

Once the plan is approved, spawn one background agent per work unit using the `${AGENT_TOOL_NAME}` tool. **All agents must use `isolation: "worktree"` and `run_in_background: true`.**

For each agent, the prompt must be fully self-contained. Include:
- The overall goal
- This unit's specific task (title, file list, change description)
- Any codebase conventions
- The e2e test recipe
- The worker instructions below

## Worker Instructions

After you finish implementing the change:
1. **Simplify** — Invoke the `${SKILL_TOOL_NAME}` tool with `skill: "simplify"`
2. **Run unit tests** — Run the project's test suite
3. **Test end-to-end** — Follow the e2e test recipe
4. **Commit and push** — Commit all changes, push the branch, create a PR
5. **Report** — End with a single line: `PR: <url>`

## Phase 3: Track Progress

After launching all workers, render an initial status table:

| # | Unit | Status | PR |
|---|------|--------|----|
| 1 | <title> | running | — |
| 2 | <title> | running | — |

As background-agent completion notifications arrive, update the table with updated status (`done` / `failed`) and PR links.

When all agents have reported, render the final table and a one-line summary (e.g., "22/24 units landed as PRs").
```

## 中文翻译

```markdown
# Batch：并行工作编排

你正在协调整个代码库的大规模可并行变更。

## 用户指令

${instruction}

## 阶段 1：研究与规划（计划模式）

调用 `${ENTER_PLAN_MODE_TOOL_NAME}` 进入计划模式，然后：

1. **理解范围。** 启动一个或多个子 agent（前台运行——你需要它们的结果），深入研究这条指令涉及的内容。找出所有需要变更的文件、模式和调用点。理解现有规范，确保迁移一致。

2. **分解为独立单元。** 将工作分解为 5-30 个自包含单元。每个单元必须：
   - 可以在隔离的 git worktree 中独立实现（与兄弟单元无共享状态）
   - 可以独立合并，无需依赖其他单元的 PR 先落地
   - 大小大致均匀（拆分大单元，合并小单元）

   根据实际工作调整数量：少量文件 → 接近 5 个；大量文件 → 接近 30 个。优先按目录或模块切片，而非任意文件列表。

3. **确定 e2e 测试方案。** 弄清楚 worker 如何端到端验证变更——不只是单元测试通过。寻找：
   - `claude-in-chrome` skill 或浏览器自动化工具（UI 变更）
   - `tmux` 或 CLI 验证工具（CLI 变更）
   - dev-server + curl 模式（API 变更）
   - 现有的 e2e/集成测试套件

   如果找不到具体的 e2e 路径，使用 `${ASK_USER_QUESTION_TOOL_NAME}` 询问用户。

4. **编写计划。** 在计划文件中包含：
   - 研究发现的总结
   - 工作单元的编号列表
   - e2e 测试方案
   - 确切的工作指令

5. 调用 `${EXIT_PLAN_MODE_TOOL_NAME}` 提交计划待批准。

## 阶段 2：启动 Worker（计划批准后）

计划批准后，使用 `${AGENT_TOOL_NAME}` 为每个工作单元启动一个后台 agent。**所有 agent 必须使用 `isolation: "worktree"` 和 `run_in_background: true`。**

每个 agent 的提示词必须完全自包含。包括：
- 总体目标
- 本单元的具体任务（标题、文件列表、变更描述）
- 任何代码库规范
- e2e 测试方案
- 以下工作指令

## 工作指令

完成变更实施后：
1. **简化** — 调用 `${SKILL_TOOL_NAME}` 工具，使用 `skill: "simplify"`
2. **运行单元测试** — 运行项目的测试套件
3. **端到端测试** — 按照 e2e 测试方案执行
4. **提交并推送** — 提交所有变更，推送分支，创建 PR
5. **报告** — 以单行结尾：`PR: <url>`

## 阶段 3：跟踪进度

启动所有 worker 后，渲染初始状态表：

| # | 单元 | 状态 | PR |
|---|------|--------|----|
| 1 | <标题> | 运行中 | — |
| 2 | <标题> | 运行中 | — |

当前台 agent 完成通知到达时，用更新后的状态（`done` / `failed`）和 PR 链接重新渲染表格。

所有 agent 报告完成后，渲染最终表格和一行动态总结（例如："22/24 个单元落地为 PR"）。
```

## 价值解析

### 核心理念：三阶段工作流

```
阶段 1: 研究规划
├── 理解范围（子 agent 研究）
├── 分解任务（5-30 个独立单元）
├── 确定测试方案
└── 编写计划文件

阶段 2: 并行执行
├── 每个单元一个 git worktree
├── 完全自包含的提示词
└── 后台运行

阶段 3: 进度跟踪
├── 实时状态表
├── PR 链接追踪
└── 最终汇总报告
```

### 关键设计原则

| 原则 | 说明 |
|------|------|
| **隔离性** | 每个单元独立 worktree，无共享状态 |
| **可合并性** | 每个单元的 PR 可独立合并 |
| **自包含** | agent 提示词包含所有上下文 |
| **进度透明** | 实时表格追踪所有单元 |

### 对 AI 编程的启发

1. **大规模任务分解**：遇到大型重构/迁移，先分解再并行
2. **独立验证**：每个变更单元都需要自己的测试方案
3. **进度可视化**：让用户实时看到所有任务的进度
