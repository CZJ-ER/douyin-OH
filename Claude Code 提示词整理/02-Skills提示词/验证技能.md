## 原版英文

```markdown
# Verify

Verify a code change does what it should by running the app.

## Step 1: Understand the Change

Start by reading git diff (or `git diff HEAD` if there are staged changes). Understand:
- What was changed
- Why it was changed
- What the intended behavior is

## Step 2: Identify How to Run

Find out how to run the application or tests:
- Check package.json for scripts
- Look for README or setup instructions
- Check for Makefile or other build tools

## Step 3: Plan the Verification

Plan specific checks to verify the change works:
1. What should happen when you run the app?
2. What should happen when you run the tests?
3. What edge cases should you test?

## Step 4: Execute Verification

Run the application and/or tests. Document:
- Command executed
- Actual output
- Whether it matches expected behavior

## Step 5: Report Results

Report verification results clearly:
- What was verified
- What passed
- What failed (if anything)
- Any issues found

## Important

- Don't just read the code and assume it works
- Actually run and test the changes
- If tests exist, run them
- If there's a UI, interact with it
- Report actual output, not assumptions
```

## 中文翻译

```markdown
# Verify

通过运行应用来验证代码变更是否符合预期。

## 步骤 1：理解变更

首先读取 git diff（或 `git diff HEAD` 是否有暂存的变更）。理解：
- 变更了什么
- 为什么变更
- 预期的行为是什么

## 步骤 2：确定如何运行

找出如何运行应用程序或测试：
- 检查 package.json 中的 scripts
- 查找 README 或设置说明
- 查找 Makefile 或其他构建工具

## 步骤 3：规划验证

规划具体的检查来验证变更是否有效：
1. 运行应用时应该发生什么？
2. 运行测试时应该发生什么？
3. 应该测试哪些边界情况？

## 步骤 4：执行验证

运行应用程序和/或测试。记录：
- 执行的命令
- 实际输出
- 是否符合预期行为

## 步骤 5：报告结果

清晰地报告验证结果：
- 验证了什么
- 什么通过了
- 什么失败了（如果有的话）
- 发现的问题

## 重要提示

- 不要只是读代码然后假设它能工作
- 实际运行和测试变更
- 如果有测试，运行它们
- 如果有 UI，与它交互
- 报告实际输出，不是假设
```

## 价值解析

### 验证流程图

```
git diff
    ↓
理解变更内容
    ↓
确定运行方式
    ↓
规划检查点
    ↓
实际执行验证
    ↓
报告结果
```

### 核心原则

| 原则 | 说明 |
|------|------|
| **不要假设** | 代码能 work 不代表真的能 work |
| **实际运行** | 运行测试，不要只读代码 |
| **记录输出** | 记录实际输出用于报告 |
| **边界测试** | 不要只测 happy path |

### 对 AI 编程的启发

1. **执行优于阅读**：读代码不能证明代码能工作
2. **文档化验证**：记录命令和输出作为证据
3. **完整报告**：不只是说"通过"，要说明验证了什么
