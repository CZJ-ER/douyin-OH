## 原版英文

```markdown
# /stuck — diagnose frozen/slow Claude Code sessions

The user thinks another Claude Code session on this machine is frozen, stuck, or very slow. Investigate and post a report to #claude-code-feedback.

## What to look for

Signs of a stuck session:
- **High CPU (≥90%) sustained** — likely an infinite loop
- **Process state D (uninterruptible sleep)** — often an I/O hang
- **Process state T (stopped)** — user probably hit Ctrl+Z by accident
- **Process state Z (zombie)** — parent isn't reaping
- **Very high RSS (≥4GB)** — possible memory leak
- **Stuck child process** — a hung git, node, or shell subprocess

## Investigation steps

1. **List all Claude Code processes**:
   `ps -axo pid=,pcpu=,rss=,etime=,state=,comm=,command= | grep -E '(claude|cli)' | grep -v grep`

2. **For anything suspicious**, gather more context:
   - Child processes: `pgrep -lP <pid>`
   - If high CPU: sample again after 1-2s to confirm
   - If a child looks hung, note its full command line

3. **Consider a stack dump** (advanced, optional):
   - macOS: `sample <pid> 3`

## Report

**Only post to Slack if you actually found something stuck.** If every session looks healthy, tell the user directly.

If you found a stuck/slow session, post to **#claude-code-feedback** with a two-message structure:

1. **Top-level message** — one short line: hostname, version, symptom
2. **Thread reply** — full diagnostic dump with PID, CPU%, RSS, state, diagnosis

## Notes
- Don't kill or signal any processes — this is diagnostic only
- If the user gave an argument (e.g., a specific PID), focus there first
```

## 中文翻译

```markdown
# /stuck — 诊断冻结/慢速的 Claude Code 会话

用户认为此机器上的另一个 Claude Code 会话冻结、卡住或非常慢。调查并向 #claude-code-feedback 发布报告。

## 需要注意的迹象

卡住会话的迹象：
- **高 CPU（≥90%）持续** — 可能是无限循环
- **进程状态 D（不可中断睡眠）** — 通常是 I/O 挂起
- **进程状态 T（已停止）** — 用户可能误按了 Ctrl+Z
- **进程状态 Z（僵尸）** — 父进程没有回收
- **非常高 RSS（≥4GB）** — 可能是内存泄漏
- **卡住的子进程** — git、node 或 shell 子进程挂起

## 调查步骤

1. **列出所有 Claude Code 进程**：
   `ps -axo pid=,pcpu=,rss=,etime=,state=,comm=,command= | grep -E '(claude|cli)' | grep -v grep`

2. **对于任何可疑的**，收集更多上下文：
   - 子进程：`pgrep -lP <pid>`
   - 如果高 CPU：1-2 秒后再次采样确认
   - 如果子进程看起来挂起了，记录其完整命令行

3. **考虑堆栈转储**（高级，可选）：
   - macOS：`sample <pid> 3`

## 报告

**只有在你真的发现问题时才发布到 Slack。** 如果每个会话看起来都正常，直接告诉用户。

如果你发现了卡住/慢速的会话，使用两条消息的结构发布到 **#claude-code-feedback**：

1. **顶层消息** — 一行简短：主机名、版本、症状
2. **线程回复** — 完整诊断转储，包含 PID、CPU%、RSS、状态、诊断

## 注意事项
- 不要杀死或发信号给任何进程——这只是诊断
- 如果用户给了参数（如特定 PID），先聚焦那里
```

## 诊断检查清单

| 指标 | 阈值 | 可能原因 | 进一步检查 |
|------|------|---------|-----------|
| CPU | ≥90% 持续 | 无限循环 | 两次采样确认 |
| State D | 任意 | I/O 挂起 | 检查磁盘/网络 |
| State T | 任意 | Ctrl+Z 误触 | - |
| State Z | 任意 | 父进程未回收 | 检查父 PID |
| RSS | ≥4GB | 内存泄漏 | 检查堆 |

### 进程状态代码

```
D = Uninterruptible sleep (usually I/O)
R = Running or runnable
S = Interruptible sleep (waiting for an event)
T = Stopped by job control signal
Z = Zombie (terminated but not reaped by parent)
```

## 价值解析

### 诊断流程

```
用户报告卡住
    ↓
扫描所有 Claude 进程
    ↓
对可疑进程深入检查
    ↓
    ├── 高 CPU？→ 再次采样确认
    ├── 异常状态？→ 记录状态和诊断
    └── 子进程挂起？→ 记录命令行
    ↓
发布报告（仅当发现问题）
```

### 报告设计亮点

| 设计 | 说明 |
|------|------|
| **两条消息结构** | 顶层简洁，详情在回复中 |
| **只报告问题** | 健康时直接告知，不发 Slack |
| **不采取行动** | 只诊断，不修复 |

### 对 AI 编程的启发

1. **量化诊断**：用具体指标（CPU%、RSS、state）而非主观描述
2. **分层报告**：简报 + 详情分离
3. **克制干预**：诊断 ≠ 修复，要谨慎
