## 原版英文

```markdown
# Memory Review

## Goal

Review the user's memory landscape and produce a clear report of proposed changes, grouped by action type. Do NOT apply changes — present proposals for user approval.

## Steps

### 1. Gather all memory layers

Read CLAUDE.md and CLAUDE.local.md from the project root (if they exist). Your auto-memory content is already in your system prompt — review it there. Note which team memory sections exist, if any.

**Success criteria**: You have the contents of all memory layers and can compare them.

### 2. Classify each auto-memory entry

For each substantive entry in auto-memory, determine the best destination:

| Destination | What belongs there | Examples |
|---|---|---|
| **CLAUDE.md** | Project conventions and instructions for Claude that all contributors should follow | "use bun not npm", "API routes use kebab-case", "test command is bun test" |
| **CLAUDE.local.md** | Personal instructions for Claude specific to this user | "I prefer concise responses", "don't auto-commit", "run tests before committing" |
| **Team memory** | Org-wide knowledge that applies across repositories | "deploy PRs go through #deploy-queue", "staging is at staging.internal" |
| **Stay in auto-memory** | Working notes, temporary context | Session-specific observations, uncertain patterns |

**Important distinctions:**
- CLAUDE.md and CLAUDE.local.md contain instructions for Claude, not user preferences for external tools (editor theme, IDE keybindings, etc.)
- Workflow practices are ambiguous — ask the user whether they're personal or team-wide
- When unsure, ask rather than guess

### 3. Identify cleanup opportunities

Scan across all layers for:
- **Duplicates**: Auto-memory entries already in CLAUDE.md → propose removing
- **Outdated**: CLAUDE.md entries contradicted by newer auto-memory → propose updating
- **Conflicts**: Contradictions between layers → propose resolution

### 4. Present the report

Output a structured report grouped by action type:
1. **Promotions** — entries to move, with destination and rationale
2. **Cleanup** — duplicates, outdated entries, conflicts
3. **Ambiguous** — entries where you need the user's input
4. **No action needed** — brief note on entries that should stay put

## Rules

- Present ALL proposals before making any changes
- Do NOT modify files without explicit user approval
- Do NOT create new files unless the target doesn't exist yet
- Ask about ambiguous entries — don't guess
```

## 中文翻译

```markdown
# 记忆审查

## 目标

审查用户的记忆格局，生成按操作类型分组的变更建议清晰报告。不要直接应用变更——提交建议待用户批准。

## 步骤

### 1. 收集所有记忆层

从项目根目录读取 CLAUDE.md 和 CLAUDE.local.md（如果存在）。你的自动记忆内容已在系统提示词中——在那里审查。注意存在的团队记忆部分（如果有）。

**成功标准**：你有了所有记忆层的内容并可以比较它们。

### 2. 分类每个自动记忆条目

对于自动记忆中的每个实质性条目，确定最佳目的地：

| 目的地 | 适合内容 | 示例 |
|--------|---------|------|
| **CLAUDE.md** | 所有贡献者应遵循的项目规范和对 Claude 的指令 | "用 bun 不用 npm"、"API 路由使用 kebab-case"、"测试命令是 bun test" |
| **CLAUDE.local.md** | 特定用户的个人指令 | "我偏好简洁回复"、"不要自动提交"、"提交前运行测试" |
| **团队记忆** | 适用于所有仓库的组织范围知识 | "deploy PR 通过 #deploy-queue"、"staging 在 staging.internal" |
| **保留在自动记忆** | 工作笔记、临时上下文 | 特定会话的观察、不确定的模式 |

**重要区分：**
- CLAUDE.md 和 CLAUDE.local.md 包含给 Claude 的指令，不是用户对外部工具的偏好（编辑器主题、IDE 键位等不属于两者）
- 工作流实践是模糊的——询问用户它们是个人还是团队范围的
- 不确定时，问而不是猜

### 3. 识别清理机会

扫描所有层：
- **重复**：已在 CLAUDE.md 中的自动记忆条目 → 建议删除
- **过时**：被更新的自动记忆否定的 CLAUDE.md 条目 → 建议更新
- **冲突**：任意两层之间的矛盾 → 建议解决方案

### 4. 呈现报告

输出按操作类型分组的结构化报告：
1. **晋升** — 要移动的条目，含目的地和理由
2. **清理** — 重复、过时、冲突的条目
3. **模糊** — 需要用户输入的条目
4. **无需操作** — 关于应保留的条目的简要说明

## 规则

- 在进行任何变更前展示所有建议
- 未经用户明确批准不要修改文件
- 除非目标不存在，否则不要创建新文件
- 关于模糊条目要问——不要猜
```

## 价值解析

### 四层记忆架构

```
┌─────────────────────────────────────┐
│         自动记忆 (Auto-Memory)         │
│    会话中的观察、临时上下文             │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│     CLAUDE.local.md (本地)            │
│    个人偏好，仅你自己可见               │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│       CLAUDE.md (项目)                │
│    项目规范，所有贡献者可见              │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│       团队记忆 (Team Memory)          │
│    组织范围，跨所有仓库                │
└─────────────────────────────────────┘
```

### 决策流程

```
新条目 → 是什么类型？
├── 项目规范？ → CLAUDE.md
├── 个人偏好？ → CLAUDE.local.md
├── 组织知识？ → 团队记忆
├── 临时笔记？ → 保留在自动记忆
└── 不确定？ → 询问用户
```

### 对 AI 编程的启发

1. **分层管理**：不同级别的知识放在不同地方
2. **先审后改**：不要自动应用，先提出建议
3. **不确定性原则**：模糊时询问而非猜测
